## 1. 왜 이 기능을 만드는가 (IR 지표 -> 기능 매핑)

dcamp 발표에서 약속한 BM 논리: "이미 12만원 자비로 낸 크리에이터가 있고, 워크샵 전환율 30%/매출 300만원. 인앱 과금 시스템을 만들면 이 숫자를 스케일할 수 있다."

Phase 2는 이 약속을 데이터로 증명하는 구간.

| IR에서 증명할 지표 | 필요 기능 | Sprint |
|---------------------|-----------|--------|
| 유료 전환율 15%+ | Stripe 결제 시스템, 프리미엄 아이템 상점 | 4 (백엔드), 5 (프론트 라이브) |
| MRR 발생 | 결제 퍼널 + PostHog 매출 이벤트 | 5~6 |
| ARPU | 프리미엄 아이템 (추가 멤버, 프리미엄 콘셉트, 의상/헤어 팩) | 5 |
| 가입자 수 | 소셜 로그인 + anonymous 마이그레이션 | 4 |
| 유저 유입 퍼널 | 공유 기능 (크리에이터 콘텐츠 -> 바이럴 -> 신규 가입 -> 과금) | 5 |
| 리텐션/인게이지먼트 | 다른 그룹 탐색 + 팔로우 (무료 3개/프리미엄 무제한) | 5 |
| PMF ("매우 실망") 40%+ | UT 인터뷰 10명+ (과금 동기, 가격 민감도 중심) | 5~6 |

## 12. 영욱 확인 필요 사항

Phase 2 개발 시작 전 영욱과 확인이 필요한 기술 결정:

1. **Stripe 연동 세부 방식**: 웹 결제 플로우 (Stripe Checkout vs Elements), 웹훅 구성
2. **3주 내 과금 시스템 개발 가능 범위**: Sprint 4 백엔드 + Sprint 5 프론트 일정 현실성
3. **AI 이미지 재생성 기술 페이저빌리티**: 의상/헤어 변경이 현재 파이프라인으로 가능한지
4. **프리미엄 아이템 중 Phase 2 구현 범위**: 최소 추가 멤버 + 프리미엄 콘셉트, 의상/헤어 팩은 가능 시 추가

## 14. 커뮤니케이션 규칙

### 14.1 PR 리뷰

- 문서 PR: 리뷰 요청 후 **24시간 이내** 리뷰 완료
- 코드 PR: 리뷰 요청 후 **24시간 이내** 리뷰 완료

### 14.2 차단 요인 (Blocker) 처리

- 차단 요인 발생 시 **즉시** Slack 공유
- Sprint Review까지 기다리지 않기
- 24시간 내 해결 안 되면 소연에게 에스컬레이션

### 14.3 문서 작업

- 기능 명세서, ERD, API 명세서는 모두 aidol-docs PR로 관리
- ERD/API 변경은 반드시 영욱 리뷰 필수
- 기능 명세서는 영욱 리뷰 후 관련 개발자 공유
