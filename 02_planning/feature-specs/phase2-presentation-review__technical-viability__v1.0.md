# Phase 2 기획 - 기술 개발 실행 타당성 평가

**작성**: 김영욱 (개발 리드) | **평가일**: 2026-02-25
**대상**: Phase 2 기획 발표 (전제이, 06개 섹션)
**평가 대상 일정**: 3주 스프린트 (W1-W3)

---

## 요약

| 항목 | 평가 | 비고 |
|------|------|------|
| **전체 일정** | ⚠️ **2배 이상 필요** | 현행 계획 대비 상당한 추가 시간 필요 |
| **핵심 리스크** | 🔴 로그인 + 자동 콘텐츠 | 데이터 마이그레이션 + 인프라 비용 |
| **실행 난이도** | 중~상 | 로그인과 자동 콘텐츠 기능 복잡도 높음 |
| **비용 증가** | ⚠️ 상당 증가 | 자동 콘텐츠 백그라운드 인프라 |

**개발팀 최종 의견**: 기획 내용 자체는 타당하나, **일정과 우선순위 재검토 필수**

## 강점: 실행 타당한 부분

### 1️⃣ 프롬프트 구조화 (검증됨)

**기술적 타당성**: ✅ 높음

- **현황**: 사용자가 "인종 × 분위기 × 헤어스타일" 선택 → 백엔드에서 영어 프롬프트 매핑
- **장점**:
  - 한국어 입력의 생성 품질 저하 문제 근본적 해결
  - 구조화된 선택지 → UI/UX 간편화
  - 일관성 높은 이미지 생성 (제이님 시뮬레이션으로 검증)
- **개발 복잡도**: 낮음 (매핑 테이블 추가)
- **비용**: 무료 (DALL-E 토큰 사용)

**개발팀 검증**: "좋습니다. 이 방식으로 가세요." (영욱님)

---

### 2️⃣ UI/UX 개선 기능들

**기술적 타당성**: ✅ 중~높음

**정의** (녹취록 및 발표 자료 기반):

1. **멤버 관리 페이지** (복잡도: 낮음)
   - 사용자가 생성한 멤버들을 한눈에 볼 수 있는 페이지
   - 멤버 간 케미(관계 설정) 관리 탭 추가
   - UT 피드백: "내가 만든 그룹에 대해 정보가 너무 없어서 아쉽다"

   ![멤버 관리 페이지](./phase2-presentation/images/03-realism-05.png)

2. **좋아요 인터랙션** (복잡도: 중간)
   - 그룹 카드, 하이라이트, 멤버 프로필에 좋아요 기능 추가
   - 사용자 간 상호작용으로 플랫폼 활성화 목표
   - 기획팀 의견: 사람들이 혼자만 있는 게 아니라는 커뮤니티 느낌 강화
   - **ERD 변경 필요**: 새 `likes` 테이블 추가

   ![인터랙션 추가](./phase2-presentation/images/03-realism-08.png)

3. **그룹 완성 후 플로우 변경** (복잡도: 낮음)
   - 현재: 그룹 완성 → "프로필 공유" 또는 "다른 그룹 만들기" 선택지
   - 변경안: 그룹 완성 → 자동으로 그룹 페이지로 이동

   ![플로우 변경 및 콘텐츠 추가](./phase2-presentation/images/03-realism-02.png)

4. **그룹 생성 직후 자동 콘텐츠 생성** (복잡도: 높음)

   사용자가 그룹을 생성하면 **운영자 개입 없이** 다음 콘텐츠가 자동으로 생성됩니다:

   a) **피드 콘텐츠**: 멤버별 인사 포즈 이미지 (멤버 수만큼)
   b) **하이라이트 콘텐츠**: 그룹 하이라이트 이미지 + 채팅 (멤버 여러 명 등장)

   - 현재: 프리셋 그룹에만 콘텐츠 존재
   - 변경안: 사용자 생성 그룹에도 즉시 자동 생성
   - 목표: 그룹 완성 직후 "살아있다"는 느낌 제공
   - **이 기능이 영욱님 우려의 핵심** (Kubernetes + 백그라운드 큐 필수)

   ![그룹 생성 직후 자동 콘텐츠 - 피드/하이라이트](./phase2-presentation/images/03-realism-07.png)

## 약점: 실행 리스크 및 우려사항

### 🔴 Critical: 데이터 마이그레이션 (로그인)

**현재 구현**:
- `aidols`, `messages` 테이블이 `anonymous_id` 필드 보유
- 로그인 시스템: 없음
- 데이터 마이그레이션 로직: 없음

**로그인 도입 시 필수 사항**:
- 익명 사용자로 생성한 데이터 → 로그인 사용자 계정으로 이관
- 데이터 정합성 검증 (중복, 손실)
- 마이그레이션 UX: 별도 "연결하기" 버튼으로 사용자 선택 (자동 실행 아님)

**개발팀 의견** (영욱님):
> "로그인 자체를 구현하는 게 문제라기보다는 기존의 데이터를 다 마이그레이션을 해야 되는 그런 문제"

**결론**: 복잡도 높음

---

### 🔴 Critical: 자동 콘텐츠 생성 (인프라 복잡도)

**현재 aidol 구현**:
- Background job queue 시스템: 없음
- Image generation: 동기식 (Gemini API 직접 호출)

**자동 생성 구현 시 필요**:
- Celery 같은 분산 작업 큐 시스템 도입
- Celery 운영 = Kubernetes 필요 (또는 동등 수준의 오케스트레이션)
- 상태 관리 (생성 중/실패/재시도)
- 인프라 비용 증가

**개발팀 의견** (영욱님):
> "자동으로 생성되는 게 정말 필요한 건지 ... 인프라를 갖추기 위해서 비용이 들거든요. ... 지금은 서버 비용이 월 몇 만원 수준에서 커버할 수 있는데 그거는 더 클 것 같아요."
>
> "이미지 생성 비용이 아니라, 백그라운드 테스크를 지속적으로 돌려야 되기 때문에 그 인프라를 말씀드린 것"

**인프라 비용 분석**:
- Kubernetes 클러스터 운영 비용: VM 인스턴스 필수 (최소 $30-50/월)
- 백그라운드 작업 큐 (Celery worker): 추가 VM 인스턴스
- 이미지 생성: 외부 API 비용 (Gemini API 요청 기반, compute와 별개)
- 현재 예상: 월 수만원 수준 → 월 수십만원대 이상으로 증가

**결론**: 복잡도 높음 (Kubernetes 필수) + 운영 인프라 비용 상당 증가

---

### 🟠 High: 3주 일정의 현실성

**현행 계획**:

```
W1: PO 버그 + 기초
W2: 로그인 + 세션
W3: 실재감 기능
```

**개발팀 평가**:

| 항목 | 복잡도 | 현황 일정 | 부담도 |
|------|--------|----------|--------|
| **로그인 + 데이터 마이그레이션** | 높음 | W2 | 🔴 심각 |
| **자동 콘텐츠 생성** (피드+하이라이트) | 높음 | W3 | 🔴 불가능 |
| **프롬프트 구조화** | 낮음 | 미배정 | ✅ 가능 |
| **UI 개선** (멤버관리/플로우) | 낮음 | W3 | ✅ 가능 |
| **좋아요 인터랙션** | 중간 | W3 | ⚠️ ERD 변경 필요 |

**결론**:
- 프롬프트 구조화, 기본 UI 개선: 3주 가능
- 로그인 + 데이터 마이그레이션: 현행 일정 충분하지 않음
- 자동 콘텐츠 생성 (피드+하이라이트): 현행 일정으로는 불가능

## 기술팀 최종 의견

### 종합 평가 (영욱님)

> **"상당히 부담이 될 것 같습니다."**
>
> 지금 봤을 때 일정은 **2배 정도 더 잡아야 될 것** 같습니다.
> 데이터 마이그레이션과 자동 콘텐츠 생성이 가장 복잡한 부분입니다.

### 기획팀이 검토해야 할 항목

**1. 로그인 도입 필요성**
- 현재: 익명 사용자 방식
- 도입 시: 데이터 마이그레이션 필수 (복잡도 높음)

**2. 자동 콘텐츠 생성 필요성**
- 현재: 백그라운드 큐 없음
- 도입 시: 인프라 구축 필요 (비용 증가)

**3. 실재감 강화 기능의 우선순위**
- 낮은 복잡도: 프롬프트 구조화, UI 개선 (멤버관리, 좋아요, 플로우)
- 높은 복잡도: 자동 콘텐츠 생성 (피드+하이라이트), 데이터 마이그레이션

## 기획 용어 검토 (공식 문서 작성 시)

### 문제점

**1. "프리셋"은 설정 용어**
- Preset = 미리 저장된 설정값 (마우스 감도, 그래픽 옵션)
- 게임 업계에서 "콘텐츠"를 지칭할 때는 사용하지 않음

**2. "유저 메이드"는 플랫폼 공식 문서에서 사용하지 않음**
- Epic: "creator-made" (creator + made)
- Unity/Roblox/Microsoft: "user-generated" (user + generated)
- AIoIA: "user-made" = 플랫폼 공식 문서 표준 아님

### 주요 플랫폼 공식 용어

| 구분 | Fortnite | Roblox | Unity | AIoIA 현재 |
|------|----------|---------|-------|-----------|
| **운영팀 제작** | Epic-made | Official | — | 프리셋 |
| **사용자 제작** | Creator-made | User-Generated (UGC) | User-Generated Content | 유저 메이드 |

**출처**:
- Fortnite: [공식 용어집](https://dev.epicgames.com/documentation/en-us/fortnite/fortnite-glossary) "Epic-made games", [About](https://www.epicgames.com/site/en-US/about) "creator-made experiences"
- Roblox: [Creator Hub](https://create.roblox.com/docs/en-us) "Official", "UGC"
- Unity: [UGC 문서](https://docs.unity.com/ugs/manual/user-generated-content/manual/overview) "User-Generated Content"

### 권장 용어 (투자 유치 문서, API 명세 등)

| 현재 | 권장 (한국어) | 권장 (영어) |
|------|-------------|------------|
| 프리셋 그룹 | **AIoIA 제작** 또는 **공식 그룹** | AIoIA-made / Official |
| 유저 메이드 | **사용자 생성** | User-Generated |
