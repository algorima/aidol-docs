# AIdol Docs 문서 작성 가이드

## 프로젝트 구조

- `01_research/`: 시장 조사, 사용자 리서치
- `02_planning/`: PRD, 기능 명세서, 스프린트 계획
  - `feature-specs/`: Sprint별 기능 명세서
  - `prd/`: 제품 요구사항 정의서
- `03_design/`: 디자인 시스템, UI/UX 가이드
- `04_development/`: ERD, API 명세서
- `05_ai-llm/`: AI/LLM 관련 실험 및 프롬프트
- `06_ir-master/`: 투자 유치 관련 문서
- `templates/`: 문서 템플릿

## 문서 품질

- 한 문서는 하나의 주제만 다룸
- 변경 시 배경/이유를 반드시 명시
- 기존 문서와의 충돌 여부를 확인 후 작성
- 다른 Sprint/섹션에서 같은 기능을 다르게 설명하고 있지 않은지 검증
- 기술 용어는 첫 등장 시 간단히 설명

## 문서 포맷

- 번호 체계: 계층적 번호 사용 (1. 1.1 1.1.1)
- 문서에 이모지 사용 금지
- 수평선(---) 구분은 기존 문서 포맷을 따름
- 기능 명세서 표 형식: `| 항목 | 내용 |` 테이블 사용

## Pull Request 작성 가이드

### Atomic PR 원칙

- 1 PR = 1 변경 주제 (하나의 기능 명세 추가, 하나의 섹션 수정 등)
- 5분 안에 읽을 수 있는 분량 유지
- 큰 변경은 여러 PR로 쪼개서 올리기

### 제목

- 형식: `docs(<scope>): <한국어 제목>`
- scope 예시: `feature-spec`, `erd`, `api-spec`, `prd`, `research`
- 72자 이내

### 설명 구조

1. **요약** - 무엇을 변경했는지 1-2문장
2. **배경** - 이 변경이 왜 필요한지 (회의 결정, 기획 변경 등)
3. **변경 사항** - 추가/수정/삭제된 내용 나열
4. **논의 필요 (선택)** - 팀원 의견이 필요한 부분
5. **관련 문서** - 영향받는 다른 문서나 PR 링크

### 충돌 확인 필수

- 기능 명세서 수정 시, 같은 기능을 다루는 다른 문서(ERD, API 명세, 다른 Sprint 명세서)와 일관성 확인
- 충돌이 발견되면 PR 설명에 명시하고 팀원 의견 요청

## YC 관련 문서

## 네이밍

- 파일명: `kebab-case.md` (예: `aidol-feature-spec-sprint2.md`)
- 레거시 파일명(`20260204__...__v0.1__soyeon.md`)은 PR #9 머지 후 간소화 예정

## 다국어

- 문서는 기본적으로 한국어로 작성
- 기술 용어는 영어 원문 병기 가능 (예: "하이라이트(highlight)")

## 수정안 제안 시 필수 사항

### Before/After 비교 필수

문서나 디자인 수정을 제안할 때는 **반드시 현재 상태를 먼저 확인**하고 비교 문서를 작성해야 함:

1. **현재 상태 확인**
   - Figma MCP로 스크린샷 확인
   - 파일 내용 직접 읽기
   - 사용자에게 현재 상태 먼저 보여주기

2. **비교 문서 작성**
   - `현재 vs 수정안 비교.md` 형식
   - 각 항목별 Before/After 명확히
   - 왜 변경하는지 이유 설명
   - 개선 효과 구체적으로

3. **설명 구조**
   ```markdown
   ### 항목명

   **현재:**
   [현재 내용]

   **문제점:**
   - ❌ 문제 1
   - ❌ 문제 2

   **수정안:**
   [제안 내용]

   **개선 효과:**
   - ✅ 개선 1
   - ✅ 개선 2
   ```

4. **예시**
   - IR Deck 수정: `ir-deck-current-vs-proposed.md`
   - 기능 명세 수정: `feature-spec-changes.md`

**절대 하지 말 것:**
- ❌ 현재 상태 확인 없이 수정안만 제시
- ❌ "12개 슬라이드"처럼 검증 안 된 정보 언급
- ❌ Before/After 비교 없이 바로 수정
