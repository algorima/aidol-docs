## 2. 행동 기반 관찰

| 시점 | 행동 | 의미 |
|------|------|------|
| 탐색 초반 | 등급 체계(a-f)를 인지했으나 의미를 이해하지 못함. "등급은 뭐지"라고 질문 | 등급/능력치 시스템에 대한 온보딩 설명 부재. 프듀 a-f 같은 레퍼런스가 없으면 직관적이지 않음 |
| 캐스팅 탐색 중 | 이미 계약완료된 연습생이 너무 많다고 언급. "사람이 너무많아요" | 인기 연습생이 선점되어 선택지가 제한됨. 후발 사용자의 캐스팅 경험 품질 저하 |
| 문제 직면 시 | "이런경우엔 어캐는거람..?"으로 대안을 질문 | 계약완료된 연습생만 남았을 때의 사용자 경로가 불명확 |

| 태도 | "피드백러의 마음으로" 살펴보겠다고 표현 | 서비스에 대한 호의적 태도. 적극적으로 피드백 의지 있음 |

## 3. 가설 검증 결과

**가설**: 캐스팅 과정에서 사용자가 몰입하여 연습생을 탐색하고 선택할 수 있는가

**판정: 부분 검증 (캐스팅 완료 전 피드백 단계)**

- 등급 체계 이해: 실패 (별도 설명 후 이해)
- 캐스팅 풀 만족도: 낮음 (인기 연습생 선점, A등급 부재)
- 직접 생성 기능 발견: 자발적 발견 실패 (안내 후 인지)
- 캐스팅 몰입 여부: 미도달 (탐색 장벽 단계에서 멈춤)

## 4. 주요 이슈 분석

### 4.1 등급 체계 직관성 부족

- 근거: "등급은 뭐지", "a가 잘하는거잖아..?" 등 질문
- 등급이 능력치를 의미한다는 것을 자체적으로 파악하지 못함
- 프로듀스 101 등의 레퍼런스 없이는 a-f 등급 체계가 직관적이지 않음

### 4.2 연습생 풀 고갈 문제

- 근거: "사람이 너무많아요", "남은사람들이 X별로없구", "a등급이 너무없음"
- 인기 연습생이 선점되면 후발 사용자의 캐스팅 경험 품질이 급격히 저하
- 선택지 부족이 곧 캐스팅 몰입 실패로 직결

### 4.3 직접 생성 기능 가시성

- 근거: 안내 전까지 "이런경우엔 어캐는거람..?"으로 막힘
- 연습생을 직접 생성할 수 있다는 기능이 UI에서 충분히 노출되지 않음
- 이 기능이 풀 고갈 문제의 핵심 해결책임에도 사용자가 발견하지 못함

## 6. 추가 관찰: UX/기능 구조 이슈

### 6.1 UX 문구 문제

- 등급(a-f) 체계에 대한 인앱 설명/툴팁 부재 → 사용자가 의미를 파악하지 못함
- "계약완료" 상태의 연습생이 리스트에 그대로 노출 → 선택 불가한 항목이 탐색 비용을 높임

### 6.2 기능 구조 문제

- 연습생 직접 생성(프롬프트) 기능의 UI 가시성 부족
- 잔여 연습생 풀이 부족할 때의 대안 부재
- 계약완료된 연습생 필터링 또는 분리 표시 기능 없음

## 7. 다음 스프린트 수정 제안 (3개)

1. **등급 체계 온보딩 추가**: 캐스팅 화면 진입 시 등급(a-f)이 능력치를 의미한다는 간단한 설명 또는 툴팁 추가. 현재 사용자가 등급의 의미를 직관적으로 파악하지 못하는 문제가 관찰됨.

2. **연습생 직접 생성 기능 가시성 개선**: 잔여 연습생이 부족하거나 계약완료 비율이 높을 때, 직접 생성 기능을 상단에 프로모션하거나 빈 상태(empty state) 안내 추가. 현재 사용자가 기능 존재 자체를 인지하지 못함.

3. **계약완료 연습생 표시 개선**: 이미 계약완료된 연습생을 필터링하거나 별도 섹션으로 분리하여, 실제 선택 가능한 연습생만 우선 노출. 선택 불가능한 항목이 리스트에 혼재되어 탐색 비용 높음.
